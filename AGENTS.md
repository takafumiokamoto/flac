# AGENTS.md

## エージェントへのルール

- ユーザーから明示的に依頼されない限り、ファイルの作成・編集を行わない。コード・テストだけでなく、テストデータ生成スクリプト、シェルスクリプトなどの補助作業も同様。エージェントは説明、設計相談、レビュー、提案に留める。
- 例外として、Markdown ドキュメント(`*.md`)の編集はこの禁止の対象外とする。

## 学習目標

このプロジェクトでは、動作する FLAC デコーダを完成させるだけでなく、**実装を通して作者が RFC 9639 を自力で読み、仕様をコードとテストへ落とし込めるようになること**を主要な目標の一つとする。

各処理について、作者が次を説明できる状態を目指す。

- RFC のどの節・表・段落が根拠なのか
- 入力のビット列やバイト列を、どのように値へ変換するのか
- どの値をなぜ検証する必要があるのか
- そのテストが仕様の何を保証しているのか
- RFC が要求することと、この実装が選んだ方針の境界はどこか

### 学習目標を支えるエージェントの振る舞い

1. **仕様を先に示す**: 実装やレビューの前に、対応する RFC 9639 の節・表・可能なら段落を示し、該当箇所の要点を短く説明する。参照は説明の近くに置き、単に文末へ参考資料を列挙するだけにしない。
2. **規定の強さを保つ**: RFC の `MUST` / `MUST NOT` / `SHOULD` / `MAY` を区別して説明する。RFC が許容している値や「decoder behavior is left unspecified」としているものを、根拠なく一律の禁止事項として扱わない。
3. **判断の種類を分ける**: 次の4種類を混同せず、必要に応じて明示する。
   - RFC に直接書かれた要件
   - RFC のビット幅や配置から導出した事実
   - Go の型、関数分割、エラーを返す場所などの実装上の判断
   - 対応ビット深度など、このプロジェクト固有の MVP 制約
4. **ビット列からコードまでをつなぐ**: 完成コードだけを示さず、「RFC 上の配置 → 具体的なバイト列・ビット列 → シフトやマスクなどの変換 → Go の値」という順序で説明する。型を選ぶ理由や境界値も説明する。
5. **段階的に支援する**: まず考え方、制約、テストケースを示し、必要なら擬似コードや小さなヒントへ進む。ユーザーが明示的に実装を依頼していない限り、完成コードを先回りして書かず、作者が自分で実装して理解する余地を残す。
6. **テストを仕様読解の道具にする**: RFC Appendix D の例や IETF testbench を可能な限りオラクルに使い、「この入力のこのビットが、期待値のこのフィールドになる」という対応を説明してからテストを提案する。
7. **レビューでも根拠を示す**: ユーザーのコードをレビューするときは、正誤だけでなく、該当コードがどの RFC 要件を実現しているか、またはどの要件に反しているかを示す。修正案を示す場合も、なぜその修正になるかをビット配置や制約まで戻って説明する。
8. **曖昧さを推測で埋めない**: 解釈に迷った場合は RFC 本文、RFC Errata、RFC 付属例、テストファイルの順に確認する。RFC に明記されていない判断は推論または実装方針であることを明示する。参照実装は最後の答え合わせに限る。
9. **一度に一概念を扱う**: 学習単位を小さく保ち、各ステップの最後に、参照した RFC、理解すべき要点、動作確認コマンドまたは通るテストを整理する。
10. **実装依頼時も説明を省かない**: ユーザーが実装を明示的に依頼した場合も、「RFC の根拠 → テストの意図 → 実装 → 動作確認」の順序を保ち、コードだけを提示しない。

## プロジェクトの背景

Pure Go の FLAC デコーダと、それを核にしたターミナル音楽プレーヤーを段階的に作る個人プロジェクト。

- 仕様の正典は **RFC 9639**。実装判断に迷ったら、ブログ記事や他実装よりも RFC を優先する。
- デコーダ本体は **標準ライブラリのみ・pure Go・`CGO_ENABLED=0` でビルド可能** を維持する。
- デコーダはライブラリパッケージとして書き、CLI は薄いラッパーにする(将来プレーヤー本体から利用するため)。
- 開発機は macOS。デコーダ自体はプラットフォーム非依存に書く(再生確認コマンドだけ macOS 前提で良い)。Windows / Linux 対応は後回し。

## 現在のフェーズ: MVP

**FLAC ファイルを読み、生 PCM バイト列に正しくデコードして吐くだけ。再生は外部ツールに任せる。**

### やる

- `fLaC` マーカー + STREAMINFO のパース。他のメタデータブロックは長さ分スキップ
- フレームデコード一式:
  - サブフレーム 4 種すべて:CONSTANT / VERBATIM / FIXED(次数 0–4)/ LPC(最大 32 次)
  - チャネル割当 4 種すべて:独立 / Left-Side / Right-Side / Mid-Side
  - チャネル数 1〜8(subset のファイル 38〜43 が 3.0ch〜7.1ch を含むため)。3ch 以上はすべて独立チャネルとしてデコードする
  - wasted bits
  - Partitioned Rice 残差(エスケープ含む)
- ビット深度 8/16/24(12/20 は同じ経路でほぼ自動対応)
- 出力:インターリーブ済み LE の生 PCM を stdout へ / `-o out.wav` で WAV。チャネル順は FLAC のチャネル割当順のままインターリーブする
- CLI: `cmd/flacdec`(名前は仮)

### やらない(後回し)

- シーク・SEEKTABLE の利用
- CRC-8 / CRC-16 の**検証**(バイトの読み飛ばし自体は行う)
- 32bit 深度(残差の 64bit 化が必要になるため別フェーズ)
- 先頭 ID3 タグのスキップ、エラー再同期
- Ogg 封入 FLAC、エンコーダ
- Go での音声出力(再生は afplay / ffplay に任せる)

## 開発の進め方(最重要)

**RFC 9639 を一次資料として、理解しながら、常に動く状態を保って小さく進める。**

1. **RFC 駆動**:各ステップの実装前に、対応する RFC 9639 のセクション番号を明示し、その要点を短く説明してから実装する。作者が仕様を理解すること自体がこのプロジェクトの目的の一つなので、説明を省いてコードだけ出さない。仕様が曖昧に感じたら推測せず、RFC 本文とテストファイルで確認する。
2. **動くものベース**:どのステップも「実行できるコマンド」または「通るテスト」で終わらせる。ステップの最後に動作確認の方法を必ず示す。
3. **小さく進める**:1 ステップ = 1 概念(例:「STREAMINFO のパース」「Rice 符号の復号」)を目安にし、一度に大きなコードを生成しない。
4. **テストを先に**:可能な限り、オラクル(MD5 / 参照デコーダ / ラウンドトリップ)を使ったテストを先に書いてから実装する。
5. **参照実装のコピーはしない**:mewkiz/flac 等は詰まったときの答え合わせにのみ使う。写経ではなく RFC から書く。
6. 大きめの設計判断(公開 API の形、パッケージ分割など)は実装前に方針を提示して合意を取る。決定したら `docs/adr/` に短く残す(後でブログ記事の下書きになる)。
7. フェーズが変わったら、このドキュメント自体を更新する。

### 進行の目安(難易度ラダー)

自家製テストデータをエンコードオプションで段階生成し、デコーダの機能解禁と 1 対 1 で対応させる:

1. `fLaC` マーカー + STREAMINFO パース → `flacdec info` でメタデータ表示
2. フレームヘッダの走査 → フレーム一覧の表示
3. CONSTANT / FIXED → `flac -l 0 --no-mid-side` でエンコードした無音・サイン波が通る
4. VERBATIM → ホワイトノイズが通る
5. LPC → デフォルト設定でエンコードしたモノラル音源が通る
6. ステレオ相関(Mid-Side 等) → `flac -8` のステレオ音源が通る
7. マルチチャネル → subset のサラウンド音源(38〜43)が通る
8. 実音源 1 曲 + IETF テストベンチの subset グループ全通過(= MVP 合格)

```
(ルート)         ライブラリ本体(package flac、モジュールルート直下)
internal/bitio/   MSB-first ビットリーダ(実装詳細なので internal に置き公開しない)

docs/adr/        設計判断の記録(ADR)
```

- ライブラリをモジュールルートに置くことで、インポートパスは `github.com/takafumiokamoto/flac` そのものになる(`flac/flac` の重複を避ける)。
- `bitio` は公開 API に含めないため `internal/` 配下に置く(後で公開するのは容易だが、逆は破壊的変更になる)。
- `testdata/` はモジュールルート直下にあると go ツールが自動で無視する。

## テスト戦略(3 段構え)

1. **MD5 オラクル**:STREAMINFO の MD5 はデコード後 PCM 全体のハッシュ。各サンプルを ceil(bps/8) バイトの LE 符号付き整数としてチャネルインターリーブした列に対して計算する(16bit なら stdout に流す列そのもの)。**MD5 が全ゼロのファイルは未設定を意味するので検証スキップ**。
2. **参照デコーダとの突合**:`flac -d --force-raw-format --endian=little --sign=signed -o ref.raw in.flac` の出力と自作デコーダの出力を `cmp` でバイト比較する。
3. **自家製ラウンドトリップ**:テスト内でサイン波・ノイズ・無音の PCM を合成し、`flac` CLI でエンコード → 自作デコーダで復元 → 元 PCM との一致を確認する。エンコードオプションは難易度ラダー(上記)に対応させる。

- 適合テストの本丸は `testdata/` の IETF testbench(CC0)。**当面のゴールは subset グループ全通過**
- **著作権のある音源はリポジトリに絶対に入れない**(手元の音楽ファイルは耳での確認専用)

## 仕様の地雷ポイント(RFC 読解の補助)

- ビットリーダは **MSB-first**
- フレームヘッダ:14bit の同期コード、UTF-8 風可変長のフレーム/サンプル番号、ブロックサイズ・サンプルレートにはエスケープ値がある
- **LPC / FIXED の予測の累積は int64 で行う**(int32 ではオーバーフローする)
- **Side チャネルは bps+1 ビット**(L/S, R/S, M/S のとき。忘れると盛大に壊れる)
- Rice パラメータのエスケープ(4bit 方式は 0b1111、5bit 方式は 0b11111)で残差が生ビット列になるケースがある
- wasted bits:フラグ 1 ビットを読み、1 なら「wasted bits の数 − 1」を unary で読む(フラグが 1 の時点で最低 1 ビットが保証されるため +1 する)。デコード後に左シフトで復元する
- 8bit の扱い:raw 出力と MD5 計算は signed、WAV の 8bit は unsigned。混同すると MD5 テストは通るのに WAV だけ壊れる(逆も然り)
- 破損・不正入力では必ずエラーで終了すること(クラッシュ・無限ループ・入力由来の巨大メモリ確保をしない)。信頼できない入力のパーサであるという自覚を持つ

## パフォーマンスについて

- MVP は正しさ最優先。マイクロ最適化はしない。
- ただし「フレームごとに新規スライスを確保して返す」形の API にはしない。呼び出し側バッファへ書き込む形を基本にし、後段のゼロアロケーション化の余地を残す。
